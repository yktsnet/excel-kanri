# マンション管理会社向け業務システム構想

## 背景・課題

現在支援しているマンション管理会社では、以下2つの業務課題が顕在化していた。

1. **書類作成の手間**:入居契約書をはじめとする定型書類を、従業員がExcelで都度手作業コピー・編集している。年間を通じて発生件数が多く、特に春の入退室シーズンには作業負荷が集中する。
2. **電話対応の負荷**:取引事業者(不動産仲介業者等)から「空室があるか」の確認電話が頻繁にかかってくる。内見申込みの前段階の確認だけで、管理会社側の電話対応コストが膨らんでいる。

いずれも「既存のExcel運用ワークフローを変えずに、自動化の恩恵だけを後付けする」という方針で設計している。SaaS導入による標準フォーマットへの移行を求めるのではなく、現場が使っているフォーマットにオーダーメイドで対応する点を重視した。

---

## システム1:書類作成自動化 + 資料検索Webアプリ

### 解決する課題

入退室時の契約書作成における入力の重複と、過去契約書を探す際の手間を解消する。

### 仕組み

**① マスター連動による契約書自動生成**

- 管理会社が運用している名簿マスター(氏名・住所・部屋番号・家賃・入居日等の列を持つExcelファイル)を共有フォルダ(Samba)上に配置。従業員は今までと同じ操作感でファイルを開き、行を追加するだけ。
- Linux側で `watchdog` によるファイル変更監視を実装。保存イベント(`IN_CLOSE_WRITE`)を検知し、前回スナップショットとの行単位diffを取ることで新規行・更新行を判定する。
- 新規行を検知した場合、`openpyxl` でテンプレート契約書を読み込み、マッピング定義(YAML)に従って該当セルへ値を転記し、年度フォルダ配下に命名規則(氏名_部屋番号_年度)で新規ファイルとして保存する。
- 退去等の更新行については、ステータス列を反映させつつ契約書側にも変更を伝播する。

**② 検索Webアプリ**


- 生成された契約書ファイル群と名簿マスターの内容を事前にインデックス化(SQLite FTS5)。

- 氏名・部屋番号等のあいまい検索で該当ファイルを検索し、ブラウザ上でプレビュー・印刷を行える。
- 検索対象は「ファイル名」ではなく「名簿マスターの構造化データ」そのものとしているため、Windowsのファイル検索(ファイル名/内容の限定的検索)よりも実用的な検索体験を実現している。
- 自然言語での問い合わせにも対応できるよう、構造化データに対するText-to-SQL方式の検索インターフェースを採用(契約書本文の自由記述欄など非構造化テキストへの対応は将来的にRAGで拡張予定)。


### 技術スタック

| レイヤー | 技術 |
|---|---|

| ファイル共有 | Samba(従業員側の操作感を変えない) |
| ファイル監視 | Python `watchdog` |
| Excel読み書き | `openpyxl` |

| マッピング定義 | YAML |
| 検索インデックス | SQLite + FTS5 |
| 自然言語検索 | Text-to-SQL(社内別プロジェクトの構成を流用) |
| Web | バックエンドAPI + フロントエンド(検索・プレビュー・印刷UI) |
| インフラ | Linux(オンプレ/VPS) |

## システム2:事業者向け空室情報配信Webアプリ


### 解決する課題

取引事業者からの空室確認電話を減らし、管理会社側の情報確認コストも下げる。

### 仕組み

- 管理会社の従業員は、今までと同様にスプレッドシートで空室状況を更新する。
- Google Apps Script(GAS)のトリガー(編集時 or 定時実行)により、スプレッドシートの内容をWebアプリ側のデータストアへ同期する。
- 取引事業者ごとにロール(松・竹・梅の3段階)を割り当て、ログイン後はロールベースアクセス制御(RBAC)により表示情報の粒度を制御する。

  - **松**(優良取引事業者):空室情報+詳細条件・内見可能時間まで開示
  - **竹**:空室の有無+一部情報を開示、詳細は問い合わせが必要
  - **梅**:空室の有無のみ、公開情報とほぼ同等の粒度
- 更新は正午(12:00)を区切りとし、画面上に「本日12:00時点の情報」と最終更新時刻を明示。これにより事業者側の情報鮮度に対する期待値を調整し、12:00以降の急ぎの確認は電話を案内する運用ルールとセットで運用する。

### 技術スタック

| レイヤー | 技術 |
|---|---|
| データ入力 | Googleスプレッドシート(従業員の操作感を変えない) |
| 連携 | Google Apps Script(onEdit / 時間主導トリガー) |
| 権限管理 | ロールベースアクセス制御(松/竹/梅) |
| Web | ログイン認証付きダッシュボード |
| インフラ | Linux(オンプレ/VPS) |

## 2システムの関係性


両システムは目的・利用者・鮮度要件が異なるため、密結合させずに独立したシステムとして構築している。

- システム1:社内向け、正確性重視(正式書類)

- システム2:対外向け、速報性重視(空室情報)

唯一の自然な接点は「部屋の入居/空室ステータス」というデータで、これは契約書生成・退去処理の完了イベントを起点に、疎結合な形(Webhook通知またはステータス同期バッチ)でシステム2側に連携する設計としている。一枚岩のシステムにせず、必要最小限のデータだけを橋渡しすることで、片方の不具合が他方に波及しないようにしている。

## システム3:自社公開サイトへのオーナー/入居者ログイン機能

既存の自社公開Webサイト(SNS等のマーケティングと連動し、紹介会社経由に依存しない直接問い合わせの受け口として運用中)に、ログイン機能を追加する構想。

### オーナー向け機能

| 機能 | 内容 |
|---|---|
| 工事スケジュール確認 | 大規模修繕・点検等のスケジュールを一覧表示 |

| 振込実行記録確認 | 管理会社経由で実行された家賃振込の完了記録を確認 |
| 入居率・空室率ダッシュボード | システム2の空室データをロール追加のみで流用し表示 |
| 収支レポート | 確定申告用の年次/月次収支レポートをPDF自動生成 |

| 修繕・原状回復履歴 | いつ・いくら・何を修繕したかの履歴記録 |
| 契約更新アラート | 管理対象物件の入居者の契約更新タイミングを事前通知 |

### 入居者向け機能

| 機能 | 内容 |
|---|---|
| 基本情報確認 | 入居日・契約更新日の確認(家賃振込関連は管理対象外。管理会社経由の業務であり責任範囲が曖昧になるため対象外とした) |
| 修繕・問い合わせフォーム | 電話対応の削減に直結する機能として優先度高 |
| 契約更新通知 | 更新時期が近づいたら通知 |
| 共用部お知らせ | メンテナンス情報等の掲示 |

### 紹介キャンペーン機能


学生入居者が多いという特性を踏まえ、紹介経由の入居を促進する機能。

- 入居者ごとに一意の紹介コードを発行(既存の名簿マスターに列を1つ追加するだけで管理可能。システム1の仕組みに乗せられる)
- 紹介コード経由のランディングページ/内覧申込フォームへの流入をトラッキング
- 内覧→入居決定までのファネルを記録
- 入居決定時に紹介者・被紹介者の双方にお祝い金を発生させ、記録として残す
- 入居者向けダッシュボードで以下を可視化
  - 紹介した人数
  - 内覧まで進んだ人数
  - 入居決定人数
  - 獲得したお祝い金合計

アナログ(チラシ)でも運用可能な施策だが、Web化により進捗の可視化自体が紹介意欲の維持に寄与すると想定している。

### 優先度

1. **紹介キャンペーン機能**:公開サイトの本来の目的(直接問い合わせの増加)に直結し、事業インパクトが最も大きい。
2. **入居者向け基本ダッシュボード+修繕依頼フォーム**:既存のマスター連動の仕組みを流用できる低コスト施策であり、電話対応削減にも寄与する。
3. **オーナー向け工事スケジュール・振込記録確認**:オーナーとの信頼関係維持に重要だが、更新頻度が低く緊急性は中程度。

---

## 設計上のポイント

- **現場のワークフローを変えない**:従業員はExcel/スプレッドシートを今まで通り編集するだけで、自動化の恩恵を受けられる。
- **段階的な移行設計**:まずファイル監視による後付け自動化から始め、将来的にはマスター編集自体をWeb入力に巻き取っていく移行ルートを想定。
- **オーダーメイド対応**:標準化されたフォーマットへの移行を前提とする一般的なSaaSと異なり、現場で既に使われているExcelフォーマットにそのまま対応する点を差別化軸としている。
- **OSS化の展望**:セルマッピング設定の汎用化(YAML化)を進めることで、自社専用ツールから他の管理会社でも使える形への一般化を検討している。
