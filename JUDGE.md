# 技術選定・設計意思決定書 (JUDGE.md)

本ドキュメントは、「マンション管理会社向け業務システム（tok-kanri）システム1」をオープンソース（OSS）プロジェクトとして開発するにあたっての、技術選定および設計上の意思決定を記録したものである。

---

## 1. 稼働環境と開発環境

### 決定事項
* **本番・運用ターゲット環境**: VPS または オンプレミス Linux サーバー
* **ローカル開発環境**: Windows + WSL2（Windows実機からWSL2のファイルシステムを直接参照する構成）

### 意思決定の背景
* **WSL2の優位性**: WSL2のファイルシステム（`\\wsl.localhost\...`）をWindowsのエクスプローラーやExcelから直接編集・保存すると、WSLのLinux側カーネルへ `inotify` イベントが正しく伝播する。これにより、追加のファイル共有サーバー（Sambaなど）を構築することなく、Windowsの操作をトリガーにしたファイル監視テストが可能になる。
* **本番想定**: 本番環境では、オンプレミス Linux または VPS上にSambaやWebDAV等の共有ディレクトリを設置し、従業員PCからその共有フォルダへ名簿マスターExcelを直接編集・保存する構成を取る。

## 2. ファイル処理・Web UIの設計

### 決定事項

* **書類生成フロー（ルートA）**:
  1. 従業員がWebアプリの入力フォームから書類種別・必要項目を送信する。
  2. FastAPIがSQLiteに記録し、`openpyxl` でテンプレートExcelに値を流し込んで `.xlsx` を生成する。
  3. Gotenberg コンテナへ HTTP リクエストを送り `.pdf` に変換して `generated/` に保存する。
* **共有フォルダ監視フロー（ルートB）**:
  1. 従業員が `shared/` フォルダへExcelを配置・保存する。
  2. サーバー側（Python `watchdog`）が変更（`IN_CLOSE_WRITE`）を検知する。
  3. Gotenberg コンテナへ HTTP リクエストを送り `.pdf` を自動更新（同名上書き）する。
* **Web UIの役割**:
  * 入力（ルートA）・閲覧（ルートA+B）・検索（ルートAのみ）の3役割を担う。
  * **Excelファイルのダウンロードボタンは提供しない**（Web上からは読み取り専用のPDF閲覧・印刷のみとする）。

### 意思決定の背景

* **PDFによるプレビューの一本化**: ExcelをブラウザでズレなくレンダリングするにはLibreOffice Online等の重厚なミドルウェアが必要だが、PDFであればブラウザネイティブのPDF描画で即座かつ正確に表示できる。
* **データ整合性の維持**: ルートAではSQLiteがデータの正（Single Source of Truth）であり、生成されたExcelをWebからダウンロードして手動修正することを防ぐ。ルートBのExcelはWindowsから直接編集可能であり、Webからのダウンロードは不要。

## 3. Webアプリケーションの技術スタック

### 決定事項

| レイヤー | 技術 |
|---|---|
| バックエンドAPI | Python / FastAPI |
| フロントエンド | React + Vite（SPA構成） |
| スタイリング | Tailwind CSS |
| UIコンポーネント | shadcn/ui |
| データベース | SQLite + FTS5 |
| PDF変換 | Gotenberg（Dockerコンテナ） |
| 認証 | JWT（passlib + python-jose） |

### 意思決定の背景

* **Python一元化によるセルフホスト容易性**: `watchdog`・`openpyxl` 等のPythonライブラリと同じランタイムにFastAPIを置くことで、`pip install` だけで済むセットアップを実現する。
* **フロントエンドの配信**: ReactのビルドアセットをFastAPIが静的配信するため、本番稼働時にNginx等の別WebサーバーやGotenbergのリバースプロキシも不要。
* **GotenbergによるPDF変換の分離**: LibreOfficeをアプリサーバーに直接インストールする代わりに、変換専用のGotenbergコンテナへHTTPリクエストで委譲する。`subprocess` 管理が不要になりアプリコードが単純化される。
* **shadcn/uiの採用**: コンポーネントを手元にコピーする方式のため自由なカスタマイズが可能。Table・Form・Dialog等の業務系コンポーネントが揃っている。

### 決定事項

## 5. 認証・権限設計

### 決定事項

* **認証方式**: JWT（アクセストークン）
* **ロール**:
  * `viewer`: ファイル一覧・PDFプレビュー・印刷・検索のみ
  * `editor`: viewerの全機能 ＋ ルートAによる書類作成（`POST /api/generate`）
* **ユーザー管理**: SQLiteの `users` テーブルで管理（既存DBに追加）。管理者ロールは設けず、ユーザー追加はシードデータまたは環境変数で行う。
* **Demoモード** (`DEMO_MODE=true`):
  * ログインページに `viewer` / `editor` タブを表示し、デモ用認証情報を自動入力する。
  * サンプルデータ（書類・ユーザー）を初期投入済みの状態で起動する。
* **通常モード** (`DEMO_MODE=false`):
  * シンプルなメール＋パスワードのログイン画面のみ表示する。
  * ロールはアカウントに紐づけ。

### 意思決定の背景

* **管理者ロール不要**: ユーザー管理はデプロイ時のシードデータで完結するため、管理UIを省略し複雑性を下げる。
* **Demoモードの独立**: OSSとして公開する際に `DEMO_MODE=true` の1変数でデモ環境が即座に用意できる設計とし、評価・採用の障壁を下げる。

## 6. リポの正体 — ツールキットと適用例の分離

### 決定事項

* 本リポの正体は「既存 Excel 運用に Web 入力・PDF 閲覧・検索を後付けするツールキット」とし、マンション管理は最初の適用例と位置づける。
* 3層構成: `packages/`（汎用モジュール。ドメイン語彙を含まない）→ `app/`（FastAPI + React による組み立て）→ `examples/mansion/`（テンプレート・マッピング・シード）。
* ドメイン固有物は `examples/` にのみ置き、クライアントの実テンプレート・実データはリポに含めない（架空様式のみ）。

### 意思決定の背景

* **ドメインの所在**: マンション管理という語彙はテンプレート Excel・YAML マッピング・フォーム定義の中にしか存在せず、機械部分（流し込み・監視・変換・検索・閲覧）は完全にドメインフリーだった。将来の別クライアント案件・他業種の OSS 利用者が欲しいのは機械部分であり、境界を最初から構造で表現した。
* **切り出しコストの最小化**: `packages/` は app を import しない一方向依存・外部依存（Gotenberg・パス・時刻）は引数注入で書く。これにより将来レジストリへ切り出す場合もディレクトリ移動程度で済む。

## 7. 配布形態 — clone リファレンス（レジストリ公開はしない）

### 決定事項

* リポ全体を「clone して使うリファレンス実装」として公開する。PyPI へのパッケージ公開は行わない。
* 利用者は clone し、`examples/` を自分の様式・マッピングに差し替えて使う。

### 意思決定の背景

* レジストリ公開には versioning・後方互換・英語ドキュメントの継続コストが伴う。汎用化のコストは利用者が現れてから払う。
* 言語・レジストリはランタイムが決める従属変数であり、「npm / pip を標準にする」という決め方をしない。本リポは openpyxl / watchdog / FastAPI という Python エコシステムの上にあるので Python、かつ利用者のビルドに組み込まれる性質の code ではないためレジストリ不要。

## 8. デモ方式 — 常設 URL を作らず VHS + 一発起動

### 決定事項

* 常設のデモ URL（ホスティング）は用意しない。
* モジュール単位のデモは VHS の `.tape` をリポに同梱し GIF を生成する。アプリ全体のデモは `docker compose up` 一発で `DEMO_MODE=true` が立ち上がることを MVP の完成条件とする。

### 意思決定の背景

* **受け手が開発者**: 配布形態が clone リファレンスである以上、デモの受け手はエンドユーザーではなく開発者（と将来の自分）。VHS + README が伝達手段として適切で、`.tape` 同梱によりデモ自体が再現可能・保守可能になる。
* **ホストコストの回避**: 常設 URL は Gotenberg コンテナ + VPS 常駐と生成 API の悪用対策を要し、ルートB（フォルダ監視）は訪問者が体験できない。「手元で1コマンドで立ち上がる」ことで評価可能性を担保し、常設 URL は必要になった時点（面談・ポートフォリオ掲載）で DEMO_MODE をそのまま VPS に置けばよい。

## 9. 検索の MVP 範囲 — FTS5 まで、Gemini は post-MVP

### 決定事項

* MVP にはキーワード検索（SQLite FTS5）と検索 UI を含める。自然言語検索（Gemini Text-to-SQL）は post-MVP とする。

### 意思決定の背景

* 解決課題「過去契約書を探す手間」はキーワード検索で解ける。SQLite 記録は生成機能で済んでおり、FTS5 テーブルと検索バーの増分は小さい。
* LLM 連携は差別化ではなく加点要素で、API キー必須になるとデモ・評価の敷居も上がる。FTS5 の上に後付けできる設計のため切り離しコストがない。

## 10. セットアップ手順の正 — venv / npm（Nix は任意）

### 決定事項

* README・ドキュメントのセットアップ手順は `python -m venv` + `pip install -r requirements.txt` + `npm install` を正とする。`shell.nix` は Nix ユーザー向けの任意ショートカットとして維持する。

### 意思決定の背景

* 想定利用者（clone する開発者）の大多数は Nix エコシステム外におり、Nix 前提の手順は導入障壁になる。shell.nix は開発機のツールチェーン（python313 / node22）を揃える入口としてのみ機能させ、日常の開発ループやドキュメントの前提にしない。
